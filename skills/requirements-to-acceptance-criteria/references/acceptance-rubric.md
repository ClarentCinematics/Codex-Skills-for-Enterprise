# Acceptance Criteria Rubric

Use this rubric to make requirements implementation-ready.

## Good Criteria

- State observable product behavior.
- Include actor, trigger, system response, and expected result when available.
- Avoid vague verbs such as improve, streamline, support, or handle without a measurable behavior.
- Include negative, permission, empty-state, error, and migration cases when relevant.

## Readiness Checks

- Owner is stated.
- Dependencies are named.
- Test scenarios exist.
- Launch constraints are described.
- Non-goals or out-of-scope boundaries are clear enough to prevent scope drift.

## Safe Language

- Use `Not stated` for missing owner, dates, dependencies, or launch constraints.
- Use "candidate acceptance criteria" when the source is incomplete.
- Do not convert assumptions into commitments.
