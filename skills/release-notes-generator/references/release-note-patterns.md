# Release Note Patterns

## Audience Modes

| Audience | Emphasis | Avoid |
| --- | --- | --- |
| Customer | value, usability, action needed | internal ticket IDs, architecture noise |
| Executive | outcomes, risk, adoption, delivery confidence | implementation detail |
| Support | customer impact, troubleshooting, known issues | vague benefit language |
| Developer/operator | migrations, compatibility, deployment notes | marketing phrasing |

## Change Buckets

- New capabilities
- Improvements
- Fixes
- Security or compliance updates
- Breaking changes
- Deprecations
- Known issues

## Language Rules

- Use past tense for shipped work.
- Use active, concrete wording.
- Prefer "Added export filtering by date range" over "Various export improvements."
- Include action required only when the source supports it.
